# IronLog v3 — Benchmark Report

Two-file commit log (metadata + payload), O(1) offset lookup, splice zero-copy write and read path.

## Environment

| Property | Value |
|---|---|
| Instance | c6a.large (2 vCPU, ~3.8 GB RAM) |
| Platform | EKS Auto Mode, Bottlerocket OS |
| Kernel | 6.12 |
| I/O runtime | io_uring (Compio) |
| Dataset (small payload) | NASA HTTP access log, July 1995 — ~1.89M records, ~150 bytes average |

### Write path cpu.stat comparison

| Version | kernel:user ratio | Notes |
|---------|-------------------|-------|
| V2 non-splice | 4.41:1 | Single file, one `write_at` per message |
| V3 non-splice (splice disabled) | 5.17:1 | Two files, two `write_at` per message |
| V3 splice (1 producer) | ~5.2:1 | Two files, splice pipe overhead |
| V3 splice (2 producers) | ~5.8:1 | Two files, splice pipe overhead |

The two-file split accounts for ~0.76:1 overhead over V2. Splice adds another ~0.6:1 on top for small payloads due to pipe machinery. Both are the cost of the O(1) read path.

The Rust heap is completely decoupled from message volume and producer count — splice never allocates a user-space buffer for the payload. `anon` does not move regardless of how many messages are written.

### Scenario 4 — Splice write, large payload (1–5 MB binary, ~88 GB total)

Single producer, 2,880 iterations × 10 binary files, ~88 GB written to a 100 Gi gp3 EBS volume.

#### cpu.stat

| Configuration | kernel:user ratio | user CPU total |
|---------------|-------------------|----------------|
| V3 NASA logs (~150 B) | ~5.3:1 | — |
| V3 binary (1–5 MB) | **22.1:1** | **5.85 seconds for 88 GB** |

The ratio inverts completely at large payload sizes. Two splice syscalls per 64 KB chunk — at 3 MB average payload that is ~48 chunks per message. The kernel DMA engine does all the data movement; user space issues the syscalls and steps aside. 5.85 seconds of Rust user-space CPU for 88 GB of data transferred.

The CPU ratio is not the relevant metric for large payloads. The absolute user CPU cost per byte written approaches zero.

#### memory.stat

| Metric | Baseline | Peak | Final |
|--------|----------|------|-------|
| anon (Rust heap) | 956 KB | 968 KB | 968 KB |
| file_dirty | 0 | 584 MB | 0 |
| pgscan_direct | 0 | 294 (total run) | 294 |
| pgmajfault | 0 | 10 | 10 |

`anon` is a flat line across 88 GB. The heap is fully decoupled from payload size and message volume.

#### Container memory limit as a write path setting

Initial runs used a 1 Gi container memory limit. At large payload sizes the page cache fills quickly, hitting the cgroup limit and triggering direct reclaim (`pgscan_direct`: 961,591 for a 10 GB run). Direct reclaim is synchronous — it blocks the write path and adds latency.

Raising the limit to 8 Gi eliminated direct reclaim entirely (`pgscan_direct`: 294 for the full 88 GB run). All reclaim shifted to background kswapd. The write path was never blocked.

For large payload workloads, the container memory limit is a write path performance setting, not just a resource budget.

## Read Path Benchmarks

### Scenario 5 — Non-splice read, no TCP_NODELAY (NASA logs)

Single consumer, one record per fetch request (batch size = 1), splice disabled, TCP_NODELAY not set.

| Metric | Value |
|---|---|
| throughput | ~23 rec/sec |
| kernel:user ratio | 4.43:1 |
| anon (Rust heap) | ~972 KB flat |
| pgscan_direct | 0 |
| pgmajfault | 0 |
| file (page cache) | 248 MB — payload file cached from sequential reads |

#### Nagle + Delayed ACK — Root Cause

Throughput of ~23 rec/sec corresponds to ~43ms per record — exactly the TCP delayed ACK timeout.

Root cause: the server sends the 23-byte metadata and ~150-byte payload in two separate `write_all` calls. When the first segment (23 bytes) arrives, the consumer's TCP stack starts a delayed ACK timer (40ms). The server's second write is held by Nagle — 23 bytes is unACKed and 150 bytes is below MSS. The payload releases only after the delayed ACK fires. This is the classic Nagle + delayed ACK deadlock.

V2 was not affected because it sent all records in one `write_all` — one large write, no two-write interaction. The cost was OOM. V3 streams one record at a time, which introduced the split.

The write path was not affected — producer frames include the full payload in a single write.

---

### Scenario 6 — Non-splice read, TCP_NODELAY (NASA logs)

Same as Scenario 5, with `set_nodelay(true)` on the server's accepted `TcpStream`.

| Metric | No TCP_NODELAY | TCP_NODELAY |
|--------|---------------|-------------|
| throughput | ~23 rec/sec | **~2,800 rec/sec** |
| kernel:user ratio | 4.43:1 | 4.33:1 |
| anon (Rust heap) | 968 KB flat | 968 KB flat |
| pgscan_direct | 0 | 0 |
| pgmajfault | 0 | 0 |

TCP_NODELAY eliminated the stall — 121x throughput improvement with identical CPU cost per operation and identical memory profile. The ratio is unchanged because the I/O pattern is identical; only the stall between messages was removed.

V2 vs V3 memory on the read path: V2 accumulated all `ConsumerResult` records in a `Vec` before sending — for 1.89M NASA log records (~150 bytes average) that is ~360 MB on the heap, enough to OOMKill. V3 streams one record at a time, keeping `anon` flat at 968 KB across the entire 1.89M record run.

### Scenario 7 — Splice read, TCP_NODELAY (NASA logs)

Single consumer, splice enabled (`feature = "splice"`, Linux), TCP_NODELAY set.

#### cpu.stat

| Version | kernel:user ratio | Notes |
|---------|-------------------|-------|
| V3 read (non-splice, TCP_NODELAY) | 4.33:1 | `read_exact_at` from file + `write_all` to TCP |
| V3 read (splice, TCP_NODELAY) | **5.65:1** | file → pipe → TCP socket, zero user-space copy |

Splice increases the kernel:user ratio on the read path for the same reason as the write path — two splice syscalls per 64 KB chunk add pipe machinery overhead that outweighs the eliminated `memcpy` for small payloads. The benefit inverts at large payload sizes.

#### memory.stat

| Metric | Non-splice | Splice (mid-run peak) | Splice (final) |
|--------|----------|-----------------------|----------------|
| anon (Rust heap) | 968 KB flat | 968–972 KB flat | 972 KB |
| kernel_stack | 80 KB | **320 KB** | 96 KB |
| slab_unreclaimable | ~159 KB | ~318 KB | ~167 KB |
| pgscan_direct | 0 | 0 | 0 |
| pgmajfault | 0 | 0 | 0 |

The Rust heap is identical between non-splice and splice. The elevated kernel memory during active splice (kernel_stack 4x, slab_unreclaimable 2x) is transient — pipe buffers are released when the consumer disconnects. At rest, splice and non-splice have nearly identical kernel memory footprints.

For small payloads (NASA logs ~150 bytes), splice costs more kernel CPU and transiently more kernel memory with no user-space memory benefit. The benefit materialises at large payload sizes.

**Key conclusions:**
- Splice costs more kernel CPU than non-splice for small payloads on both read and write paths
- Splice benefit inverts at large payloads — 22.1:1 ratio, near-zero user CPU per byte
- TCP_NODELAY is a prerequisite for read path throughput — 121x improvement, same CPU cost
- Rust heap stays flat at ~968 KB across every configuration regardless of payload size or message volume
- Container memory limit is a write path latency setting for large payloads, not just a resource cap
