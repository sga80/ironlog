# IronLog v2 — Benchmark Report

Single-node async broker. io_uring via Compio, thread-per-core, share-nothing architecture.

## Environment

| Property | Value |
|---|---|
| Instance | c6a.large (2 vCPU, ~3.8 GB RAM) |
| Platform | EKS Auto Mode, Bottlerocket OS |
| Kernel | 6.12 |
| I/O runtime | io_uring (confirmed via Compio driver log) |
| Dataset | NASA HTTP access log, July 1995 |
| Records per producer | 1,891,714 lines (full NASA July 1995 log) |
| Message format | Text frames, binary wire protocol |

io_uring was confirmed active from the Compio runtime log at server startup on Bottlerocket Kernel 6.12. Compio falls back to epoll on kernels below 5.x; this run did not fall back.

## Benchmark Scenarios

### Scenario 1 — 1 Producer, 1 Channel (sequential)

Single producer writing ~500k records to one channel. The channel is pinned to Thread-0 via `hash("nasa-alpha") % 2`.

| Metric | Value |
|---|---|
| Server CPU (millicores) | ~474m |
| Node CPU utilisation | ~95% |
| kernel:user ratio | ~5.2:1 |
| Memory (server pod) | 5 Mi |
| `nr_throttled` | 0 |

The 5.2:1 kernel:user ratio means for every microsecond spent in user space, the kernel spent 5.2µs — driven by the per-message `write_all_at` syscall still entering the kernel for each record. io_uring reduces but does not eliminate this; submission ring batching (v4) is the path to reducing it.

---

### Scenario 2 — 2 Producers, 1 Channel (serialised)

Two producer pods each writing ~500k records to the same channel (`nasa-alpha`). Because `hash("nasa-alpha") % 2 == 0` for both, both connections queue behind Thread-0. The second producer's TCP connection waits in the flume channel until the first producer finishes and disconnects. This is the known serialisation limitation documented in DESIGN_V2.md.

| Metric | Value |
|---|---|
| Server CPU (millicores) | ~571m |
| Node CPU utilisation | ~95% |
| kernel:user ratio | ~5.2:1 |
| Memory (server pod) | 5 Mi |
| `nr_throttled` | 0 |

The kernel:user ratio is identical to Scenario 1 — confirming this is sequential execution on one thread, not true parallelism. The slightly higher CPU is the overhead of the second TCP connection being open while the first runs.

---

### Scenario 3 — 2 Producers, 2 Channels (truly parallel)

Two producer pods writing to `nasa-alpha` and `nasa-beta` respectively — different channels that hash to different worker threads (Thread-0 and Thread-1). Both producers run concurrently on separate cores with no contention.

| Metric | Value |
|---|---|
| Server CPU (millicores) | ~1171m |
| Node CPU utilisation | ~112% (node saturated) |
| kernel:user ratio | ~7.7:1 |
| Memory (server pod) | 5 Mi |
| `nr_throttled` | 0 |
| Total duration | ~4 min 20 sec |
| Aggregate throughput | ~3,850 msg/sec |
| Producer skew | Both completed within 1 second of each other |

The node reaching 112% CPU (beyond its 2-core budget momentarily) confirms the producers were running in true parallel and saturating the machine. The kernel:user ratio rising from 5.2:1 to 7.7:1 under parallel load reflects proportionally more kernel time as two threads submit io_uring operations concurrently.

Memory held at 5 Mi throughout all three scenarios — the append-only, positional-write design (`write_all_at` + manual `byte_offset`) does not buffer records in user space beyond a single frame.

`nr_throttled: 0` in all runs confirms the Kubernetes CPU limits were not hit; the node was compute-bound but not cgroup-throttled.

> **Note:** The duration (~4m20s) and throughput (~3,850 msg/sec) figures for Scenario 3 were produced by a build with several bugs — incorrect offset tracking caused early termination and incomplete writes. The kernel:user ratio (7.7:1) and CPU figures remain valid as they measure CPU behaviour independently of correctness. Corrected numbers are in the TaskRunner section below.

## TaskRunner Architecture — Corrected Benchmarks

After fixing offset/byte_offset bugs and introducing the TaskRunner spawn architecture (per-channel long-lived compio tasks replacing the blocking worker loop), the same hardware was re-benchmarked with the full 1,891,714-record dataset.

### Scenario 4 — 2 Producers, 2 Channels, TaskRunner (truly parallel, correct)

| Metric | Value |
|---|---|
| Server CPU (millicores) | ~629m |
| kernel:user ratio | 4.41:1 |
| Memory (server pod) | 236 MiB |
| `nr_throttled` | 0 |
| Total duration | 8m27s |
| Records per producer | 1,891,714 |
| Aggregate throughput | ~7,460 msg/sec (3,730 per producer) |
| Producer skew | Both completed within 1 second of each other |

The 236 MiB memory reflects page cache from ~140 MB of commit log data on disk (1.9M records × 37 bytes × 2 channels), not heap. The server heap itself remains minimal — each frame is written to disk immediately with no in-memory buffering.

The CPU halving (1171m → 629m) and ratio improvement (7.7:1 → 4.41:1) come entirely from the TaskRunner spawn architecture. With the old blocking worker loop, each producer connection held the thread until disconnect. With spawned compio tasks, the scheduler interleaves tasks at await points, keeping the io_uring submission queue fuller and getting more work through per kernel trip.

---

### Scenario 5 — 2 Producers + 1 Consumer, concurrent

Same two producers as Scenario 4, with a consumer fetching from `nasa-alpha` simultaneously. The `nasa-alpha` TaskRunner serialises produce and consume connections through its internal queue.

| Metric | Value |
|---|---|
| Server CPU (millicores) | ~630m sustained, ~693m peak (consumer-only phase) |
| kernel:user ratio (produce+consume) | 4.43:1 |
| kernel:user ratio (consumer-only phase) | 2.91:1 |
| `nr_throttled` | 0 |
| Producer duration | 8m20s (both within 1 second of each other) |

The consumer-only ratio dropping to 2.91:1 reflects the different I/O pattern — reading from warm page cache and streaming to TCP is more user-space heavy than disk writes. The concurrent ratio (4.43:1) matches the produce-only baseline, confirming the consumer adds no measurable overhead to the produce path.

**Consumer OOM:** The consumer-cli accumulates all results in `Vec<ConsumerResult>` before printing. At ~3.7M records (two accumulated runs on the PVC), the pod OOMKilled after 8m39s. This is a known client-side design limitation — the server remained healthy throughout. Fix: stream results to a callback rather than buffering.

## Architecture Observations

**Share-nothing validated.** Each channel is pinned to one thread for its lifetime. The two parallel producers completed within 1 second of each other despite writing independently to separate files — no cross-thread coordination was needed.

**io_uring confirmed.** The Compio runtime selected the io_uring driver at startup (Bottlerocket Kernel 6.12 satisfies the minimum). The 5.2:1 kernel:user ratio in the single-producer baseline is the remaining syscall overhead per-message write.

**Memory efficiency confirmed.** 5 Mi across all scenarios regardless of record count. No buffering, no accumulation — each frame is written immediately at the tracked byte offset.

**Serialisation ceiling.** Two producers on the same channel cap at approximately single-producer throughput (~474m CPU, 5.2:1). True throughput scaling requires either different channels per producer or per-connection compio task spawning (tracked in DESIGN_V2.md as the v3 fix).

## Key Ratios for v4 (Batching)

The kernel:user ratio is the primary signal for v4 batching effectiveness. Each `write_all_at` call is one kernel entry regardless of payload size. Batching N records into one write call should collapse N kernel entries into one, targeting a ratio closer to 1:1.

| Scenario | kernel:user | Target after v4 batching |
|---|---|---|
| Single producer (blocking worker) | 5.2:1 | < 1:1 |
| 2 parallel producers (blocking worker) | 7.7:1 | < 1:1 |
| 2 parallel producers (TaskRunner) | 4.41:1 | < 1:1 |
| Consumer fetch | 2.91:1 | < 1:1 |

## What v2 Deliberately Does Not Solve

- Per-connection compio task spawning (concurrent producers on same channel)
- Producer batching — one syscall per N records (v4)
- Consumer offset persistence across restarts
- Segment rolling and index files
- Zero-copy consumer read path via `sendfile()` (v5)
- Replication and HA (v3)
