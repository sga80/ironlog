# IronLog v1 — Decision Log

Single-node, synchronous broker. The goal of v1 is correctness and clarity over performance.

## Wire Protocol

**Decision: Custom binary wire protocol over text-based (e.g. JSON over TCP)**
A binary protocol is more efficient to parse and closer to what a real broker uses. JSON would be easier to debug but adds parsing overhead and isn't representative of systems engineering. This is a learning project — learning the hard way is the point.

**Decision: Big-endian byte order**
Network byte order convention. Any client in any language will expect big-endian for network protocols. Little-endian is what the CPU uses internally but the wire is big-endian.

**Decision: Request type sent once per connection as a handshake, not per message**
The broker reads 2 bytes on connect to identify the client as a producer or consumer, then routes accordingly. Subsequent frames omit the request type — it is a connection-level identifier. This reduces per-message overhead and simplifies frame parsing.

**Decision: Channel name in the wire frame header, not the payload**
The broker needs to route the message before touching the payload. Putting the channel name in the header keeps routing logic clean and separate from payload interpretation.

**Decision: `length` field covers payload only, not the full header**
The header fields are fixed-size and self-describing. Only the payload length is variable and needs to be communicated explicitly.

## Commit Log

**Decision: One commit log directory per channel**
Avoids contention — writes to different channels never compete for the same file descriptor. The tradeoff is more open file descriptors and more complex HA/replication later. Accepted for v1.

**Decision: Message offsets, not byte offsets**
Byte offsets tie consumers to the physical file layout. If the on-disk format ever changes, all consumer positions become invalid. Message offsets (0, 1, 2...) are stable across format changes and more intuitive for consumers doing time-based or position-based replay.

**Decision: Broker timestamp, not producer timestamp**
The broker controls the clock. Producer timestamps require trust and clock synchronization across machines. For v1, broker timestamp is simpler and sufficient.

**Decision: Append-only file descriptors kept open**
Opening and closing a file descriptor per write would be expensive. The broker keeps one file descriptor per channel open in append mode, stored in a `HashMap<String, ChannelState>`. This avoids repeated open/close syscalls and lets the OS optimize sequential writes.

## On-Disk Record Format

```
| 8 bytes: offset | 8 bytes: timestamp | 2 bytes: payload type | 4 bytes: payload length | N bytes: payload |
```

**Decision: Store payload type on disk**
Consumers need to know how to interpret the bytes. The channel name is already in the filename, but payload type must travel with each record.

**Decision: Store payload length on disk**
Unlike in-memory structs where `Vec::len()` gives you the size, on disk there are no natural boundaries. The reader must know how many bytes to read for each record.

## Architecture

**Decision: Separate `core` library for wire protocol types**
Both the server and producer/consumer clients need to serialize and deserialize the same wire frames. Putting this in a shared `core` crate avoids duplication and ensures both sides stay in sync.

**Decision: Synchronous, single-threaded for v1**
Concurrency adds significant complexity. Understanding the synchronous baseline — how data flows, where the bottlenecks are — is essential before introducing async. v2 will introduce Tokio. The benchmark numbers from v1 will justify the async investment.

**Decision: No connection pooling in producer v1**
A single persistent connection with reconnect on failure is sufficient for v1. Connection pooling is designed for concurrent producers — premature until v2 introduces async.

## Known Limitations

- **Single producer at a time**: the server loop is single-threaded. While one producer is connected, all other producers queue in the OS TCP accept backlog. They are accepted one at a time after the current producer disconnects. This is resolved in v2 with async.

## Consumer Design

**Decision: Fetch-and-close, not tail/follow**
The server streams all available records from the requested offset, then closes the connection. The consumer reconnects if it wants more. Tailing would block the single-threaded server indefinitely, starving all other consumers. Resolved in v2 with async.

**Decision: Optional offset via flag byte**
`ConsumerFrame` carries a 1-byte flag (`0x00` = start from beginning, `0x01` = use provided offset) followed by a fixed 8-byte offset field. This keeps the frame fixed-width after the channel name — simpler to parse than a variable-length optional field.

**Decision: Single port for producers and consumers**
The request type field (`0x0001` Produce, `0x0002` Fetch) distinguishes producer from consumer traffic. A separate port adds operational complexity without protocol benefit for v1.

**Decision: ConsumerResult mirrors the on-disk record format**
The wire format sent to consumers is identical to the on-disk record format. This is intentional — in a future zero-copy read path, records can be sent directly from the page cache without transformation.

**Decision: No zero-copy for v1 consumer reads**
Records are read from disk into user space and written to the TCP stream. Zero-copy consumer reads (`sendfile()`) are deferred to v5.

## What v1 deliberately does not solve

- Concurrent connections (v2)
- Replication and HA (v3)
- Producer batching (v4)
- Zero-copy write path via `splice()` (v5)
- Consumer offset persistence across restarts
- Segment rolling and index files
